Group 1: Foundations

The five components of a search problem are: initial state, successor function, goal test, path cost, and strategy.

In a state space graph, each state appears exactly once. In a search tree, the same state can appear multiple times because there are multiple paths to reach it.

The frontier contains nodes that have been generated but not yet expanded. The explored set contains nodes that have been expanded (removed from frontier and had successors generated).

Graph search avoids revisiting states by maintaining an explored/visited set. Before expanding a node, it checks if the state is already in the explored set.

For deterministic marking, DFS and BFS generate successors in alphabetical order. For UCS and A*, ties are broken by comparing path tuples lexicographically.

Group 2: Search Behaviour

DFS uses a stack (LIFO) frontier. BFS uses a queue (FIFO) frontier. UCS uses a min-heap by g(n). A* uses a min-heap by f(n) = g(n) + h(n).

On the office graph, DFS returns path corridor - mail_room - storage - lab with cost 11.

BFS returns path corridor - office_a - lab with cost 10. BFS is optimal only when all edge costs are equal.

UCS returns path corridor - office_b - lab with cost 6. UCS is always optimal for non-negative edge costs.

In UCS, stale heap entries are handled by lazy deletion: if the popped node is already in visited, skip it.

A* expands fewer nodes than UCS because the heuristic steers the search toward the goal. On the office graph, A* expands 5 nodes versus UCS's 6.

The Euclidean heuristic h(n) = sqrt((x_n - x_g)^2 + (y_n - y_g)^2). It is admissible because straight-line distance never overestimates the actual path cost. It is also consistent because h(n) ≤ c(n,n') + h(n') for all edges.

A limitation of Euclidean distance: when edge costs include non-distance factors (e.g. tolls), the heuristic becomes a loose lower bound, so A* gains little pruning advantage over UCS.

Group 3: Planning Models

In STRIPS, each action has preconditions, add effects, and delete effects.

The transition rule is: Result(s, a) = (s - Del(a)) ∪ Add(a), provided Pre(a) ⊆ s.

move(robot, from, to) preconditions: at(robot, from), connected(from, to). If a door exists on the edge, also needs unlocked(lab_door).

pickup_keycard(robot, keycard, room) preconditions: at(robot, room), at(keycard, room), handempty(robot). Add effects: has(robot, keycard). Delete effects: at(keycard, room), handempty(robot).

unlock_door(robot, keycard, lab_door, here, there) preconditions: at(robot, here), has(robot, keycard), locked(lab_door), door_between(lab_door, here, there). Add: unlocked(lab_door). Delete: locked(lab_door).

In the lab_via_office_b plan, action 5 move(robot, office_b, lab) fails because: connected(office_b, lab) does not exist — the door connects corridor and lab, not office_b and lab.

Inserting unlock_door before action 5 doesn't help because: (1) the robot never picked up the keycard (has(robot, keycard) is false) and (2) even if unlocked, there is no edge between office_b and lab.

In solve_planning, states are canonicalised as sorted tuples of fact tuples to ensure equivalent states get the same ID and aren't revisited.

Canonical delivery needs 9 actions. Unlocked lab needs 5 actions. The difference is 4 extra actions for the keycard detour: move to office_a, pickup keycard, move back, unlock door.

Key Concepts

Admissible Heuristic
Never overestimates true cost: h(n) ≤ h*(n) for all n. Guarantees A* finds optimal solution.
Consistent Heuristic
h(n) ≤ c(n,n') + h(n') for all edges. Implies admissibility. Guarantees A* never re-opens nodes.
Lazy Deletion (UCS/A*)
When popping from heap, if node already visited, skip it. Stale entries are left in heap and ignored when encountered.
State Space vs Search Tree
State space: each state once. Search tree: same state appears multiple times (one per path). Graph search uses visited set to bridge the gap.
STRIPS Transition Rule
Result(s, a) = (s - Del(a)) ∪ Add(a), only if Pre(a) ⊆ s.
State Canonicalisation
state_id = tuple(sorted(facts)). Ensures equivalent states hash the same way for visited-set checks.
Generated vs Expanded
Generated: pushed to frontier. Expanded: popped from frontier and successors generated. generated_count ≥ expanded_count.
Frontier Peak
Maximum size of the frontier during search. Measures memory usage of the algorithm.
